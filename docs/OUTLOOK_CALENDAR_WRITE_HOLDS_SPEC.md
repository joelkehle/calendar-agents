# Spec: Working Holds via the Outlook Calendar Write Agent

Status: v2 — Codex-reviewed 2026-06-11 (8 findings incorporated; review at
`/tmp/codex-holds-spec-review.txt`, summarized in §Review log). Ready to implement.
Sanction: Joel, 2026-06-11 ("agents managing my time, including scheduling time
with me, is a KEY productivity unlock" — memory `feedback-agents-manage-joels-time`).

## Problem

`ucla-tdg-outlook-calendar-write-agent` only accepts calendar-guard quota blocks
(summary "Meeting Quota Reached" or prefix "No more meetings",
`internal/outlookcalendarwrite/validation.go: allowedSummary`). Agents that want
to book a working session with Joel must hand him an `.ics` file. Joel wants
agents to book the hold directly.

## Design

Add a second sanctioned event class, **working hold**, alongside the existing
guard block. Guard-block validation, markers, and tests are byte-for-byte
unchanged; every widening below applies only to the hold path.

### Classification first (review §4)

`handleInsert`/`handlePatch` classify the request BEFORE any hold-specific
checks: an insert is a *working hold* iff its summary starts with
`HoldSummaryPrefix = "Joel + "`; a patch targets a hold iff the EXISTING event
carries the exact hold marker block (below). Everything else flows through the
guard path exactly as today — including when `OUTLOOK_CALENDAR_WRITE_HOLD_REQUESTERS`
is unset. Regression test required: guard insert succeeds with the env unset.

### Working-hold validation (insert)

- Timed only — `start.date_time`/`end.date_time` required; all-day forms rejected.
- Duration ≥ 15 min, ≤ 2 h; start/end on the same local date (reuse `sameLocalDate`).
- Start in the future, within the next 30 days.
- Local start between 07:00 and 21:00.
- Time zone (review §7): `time_zone` must be absent or exactly `America/Los_Angeles`,
  AND each `date_time`'s numeric offset must equal that zone's offset at that
  instant. At startup the agent verifies the Windows host's local zone matches
  `DefaultTimeZone`; if not, hold writes are refused (guard writes unaffected —
  they are all-day and date-anchored).
- `show_as` must normalize to `busy`.
- Attendees / recurrence / conference fields: rejected via `ProhibitedFields` (unchanged).
- Description (agenda) must be non-empty after marker stripping.

### Conflict check (review §2 — blocker)

The writer itself MUST reject a hold insert that overlaps any existing
non-free Outlook event, **except calendar-guard quota blocks** (Joel's ruling,
2026-06-11): a candidate event is exempt from conflict iff its body contains
both exact guard ownership marker lines (`managed_by=jk-calendar-guard-agent`
and `owner_agent=ucla-tdg-outlook-calendar-write-agent`). Guard blocks exist to
stop meetings; sanctioned working holds may overlap them. Summary text is NOT
used for the exemption (spoofable by any meeting title). All other busy events
— real meetings, private appointments, other holds — still conflict. Implemented inside the PowerShell COM script (the
laptop has the calendar locally; no bus round-trip): before `Items.Add`,
enumerate events intersecting [start, end) with `BusyStatus != olFree`; if any
exist, fail with error `conflict` carrying only the overlapping busy
*time ranges* (never subjects — privacy). Patches re-run the same check against
the merged times, excluding the patched event itself.

### Ownership markers — spoofing-resistant (review §1 — blocker)

Inbound `event.description` for holds MUST NOT contain any reserved key
(`managed_by=`, `owner_agent=`, `hold_class=`); if it does, refuse — markers
are only ever generated by the agent. The agent appends the exact block:

```text
managed_by=<authenticated bus sender id (evt.From)>
owner_agent=ucla-tdg-outlook-calendar-write-agent
hold_class=working-hold
```

`managed_by` is ALWAYS the authenticated envelope sender, never request data.
`HasHoldMarker` matches the full three-line block exactly (anchored lines, not
substring `Contains`). A hold patch additionally requires the existing event's
`managed_by` to equal the requesting sender — agents can only patch their own
holds. Guard-block `HasOwnershipMarker` is untouched.

### Service-level defense parity (review §3 — blocker)

`service.go`'s PowerShell patch script currently refuses to mutate items
lacking the guard marker lines. It must be extended to accept EITHER the exact
guard ownership block OR the exact hold ownership block, with the guard check
retained verbatim. The Go layer's checks do not replace this; the COM script
remains the last line of defense.

### Patch and cancel (review §5)

State machine, exhaustively:

1. **Active hold** → *modify*: patch summary (must keep `HoldSummaryPrefix`),
   description, times (re-validated under all insert rules + conflict check).
2. **Active hold** → *cancel*: a patch whose ONLY changes are summary gaining
   the literal prefix `[CANCELLED] ` and `show_as` becoming `free`. No other
   field may change in the same patch. Time/window rules not re-enforced.
3. **Cancelled hold** (summary starts `[CANCELLED] `): ALL further patches
   refused. No un-cancel.

No delete capability is added. PowerShell maps `show_as: free` →
`BusyStatus = 0` (olFree).

### Authorization, rate limits, idempotency (review §6)

- Env `OUTLOOK_CALENDAR_WRITE_HOLD_REQUESTERS`: comma-separated allowlist of bus
  agent ids permitted hold inserts/patches. Unset/empty ⇒ hold requests refused
  (fail closed); guard path unaffected.
- Caps on LIVE (non-dry-run) successful inserts, keyed by the hold's *event
  local date* (not submission date): max 2 per requester per event-date, max 5
  globally per event-date. In-memory; restart resets — accepted because the
  conflict check independently prevents double-booking, so a reset only
  loosens spam protection, not correctness.
- Idempotency: cache bus conversation/request id → response for 24 h; a
  duplicate insert request returns the original result instead of creating a
  second event.

### Out of scope (explicitly)

- Attendee invitations, email sends, recurring holds, true deletion.
- JK-profile rollout (personal holds wait for `jk-outlook-calendar-agent`
  restore; same binary inherits the feature).
- No new state store: Outlook remains source of truth; rate/idempotency caches
  are ephemeral process memory (STATE_ARCHITECTURE.md v1.4 reviewed — no tier change).

## Implementation notes

- `contracts.go`: `HoldSummaryPrefix`, `CancelledPrefix`, `HoldClassMarker`,
  reserved-key list; requester id threaded into build funcs (`BuildHoldInsert(event, requester)`).
- `validation.go`: classification helper + hold rule set; guard path and its
  tests unmodified.
- `service.go`: conflict-check + dual-ownership acceptance in the PowerShell;
  olFree mapping; host-timezone probe at startup.
- `agent.go` (review §4): classify → (holds only) allowlist → idempotency →
  rate limit → validation → service. Structured error codes:
  `not_allowlisted`, `rate_limited`, `conflict`, `invalid_hold`, `tz_mismatch`.
- `cmd/outlook-calendar-write-agent/main.go` + `scripts/install-outlook-calendar-write-agent.ps1`
  (review §8): parse and install `OUTLOOK_CALENDAR_WRITE_HOLD_REQUESTERS`;
  document the UCLA scheduled-task value in OUTLOOK_CALENDAR_WRITE_AGENT.md.
- Dry-run default applies to holds identically.
- Tests: every rule above; guard regression suite untouched and green with the
  new env unset; spoofed-marker insert refused; cross-agent hold patch refused;
  overlap insert refused; cancel-then-patch refused; duplicate request id
  returns cached response; 3rd same-event-date hold per requester refused.

## Post-v1 additive amendments

- 2026-06-11 (SCHEDULER_TRAVEL_SPEC §3.7): idempotency-cache hits — hold and
  travel classes alike — now carry `"replayed": true` in the response; fresh
  responses omit the field. Cached events reflect state AS WRITTEN, not
  current state. No other hold behavior changed; every original hold test
  passes unmodified.
